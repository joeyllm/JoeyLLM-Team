# Contributing to JoeyLLM

Thanks for contributing to JoeyLLM.

This repository is used to organise project documentation, planning, and related technical notes. Keep contributions clear, reviewable, and easy for the rest of the team to follow.

## How to Contribute

1. Create or update a branch for your work.
2. Make focused changes on one topic at a time.
3. Check that file names, links, and formatting are consistent.
4. Open a pull request with a clear summary of what changed and why.

## Pull Request Expectations

Each pull request should include:

- a short description of the change
- the reason for the change
- any affected files or folders
- how you checked the change

If the change updates documentation only, say that explicitly in the pull request.

## Naming Conventions

Use consistent, descriptive names so files are easy to scan in GitHub.

- Prefer lowercase kebab case for new Markdown filenames: `example-file.md`
- Keep existing folder names as they are unless the team agrees on a rename
- Use clear, descriptive names instead of vague names like `notes.md` or `misc.md`
- Keep weekly files aligned with the current convention: `week-XX-report...`

## Documentation Guidelines

- Write in Markdown
- Keep pages short and split long documents by topic
- Update links when you rename files
- Keep README files accurate when folder contents or purpose change
- Prefer practical instructions over placeholders

## Code and Notebook Changes

If you add code, scripts, or notebooks:

- include a short note explaining what the code does
- keep setup steps reproducible
- avoid committing large generated outputs or datasets
- document where external data or compute resources are required

## Testing and Verification

Before submitting a pull request:

- check Markdown formatting and internal links
- confirm renamed files are reflected in references
- run relevant tests if your change affects code
- mention any tests you ran, or state clearly if no automated tests apply

For documentation-only changes, a manual review is usually enough.

## Commit Style

Use short, descriptive commit messages that explain the actual change.

Good examples:

- `Add repository contribution guidelines`
- `Rename weekly update files to weekly report files`
- `Update infrastructure docs for WireGuard access`

## Collaboration

- Keep discussion constructive and specific
- Raise uncertainties early instead of guessing
- Do not overwrite or remove another person’s work without checking first

## Questions

If you are unsure where something belongs, start by checking the relevant folder README and the root [README.md](README.md).
